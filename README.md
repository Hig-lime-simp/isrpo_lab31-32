## Краткое описание работы

В данной лабораторной работе было разработано ASP.NET Core Web API приложение для управления списком задач (Todo List) с использованием:

- **Entity Framework Core** — ORM для работы с базой данных
- **SQLite** — легковстраиваемая реляционная база данных
- **Code First подход** — создание базы данных на основе C# классов
- **LINQ** — мощные запросы к базе данных

Приложение поддерживает полный CRUD функционал, поиск, фильтрацию, пагинацию, статистику и работу со сроками выполнения задач.

## Полезные команды dotnet ef

| Команда | Описание |
|---------|----------|
| `dotnet ef migrations add Имя` | Создать новую миграцию по изменениям в моделях |
| `dotnet ef database update` | Применить все непримененные миграции к БД |
| `dotnet ef migrations list` | Посмотреть список миграций и их статус |
| `dotnet ef migrations remove` | Удалить последнюю миграцию (если не применена) |
| `dotnet ef database update ПредыдущаяМиграция` | Откатить до указанной миграции |
| `dotnet ef migrations script` | Сгенерировать SQL скрипт без применения к БД |

## Сравнительная таблица: LINQ vs SQL

| LINQ | SQL |
|------|-----|
| `.Where(t => t.IsCompleted == false)` | `WHERE is_completed = 0` |
| `.OrderBy(t => t.CreatedAt)` | `ORDER BY created_at ASC` |
| `.OrderByDescending(t => t.CreatedAt)` | `ORDER BY created_at DESC` |
| `.Take(10)` | `LIMIT 10` |
| `.Skip(20).Take(10)` | `OFFSET 20 LIMIT 10` |
| `.Count()` | `SELECT COUNT(*)` |
| `.Any(t => t.Priority == "High")` | `SELECT EXISTS(...)` |
| `.Max(t => t.Id)` | `SELECT MAX(id)` |
| `.GroupBy(t => t.Priority)` | `GROUP BY priority` |
| `.Select(t => t.Title)` | `SELECT title` |
| `t.Title.Contains("query")` | `WHERE title LIKE '%query%'` |

## Сравнительная таблица: Хранение в памяти vs EF Core + SQLite

| Концепция | Хранение в памяти (`static List<T>`) | EF Core + SQLite |
|-----------|--------------------------------------|------------------|
| **Хранение данных** | `static List<T>` в RAM | Файл `.db` на диске |
| **После перезапуска** | Данные пропадают | Данные сохраняются |
| **Поиск по условию** | LINQ to Objects | LINQ to Entities → SQL |
| **Создание структуры** | Не нужно | Миграции (`dotnet ef`) |
| **Начальные данные** | Хардкод в коде | `HasData()` в миграции |
| **Получение данных** | `list.FirstOrDefault(...)` | `await db.Table.FindAsync(id)` |
| **Добавление** | `list.Add(item)` | `db.Table.Add(item)` + `SaveChangesAsync()` |
| **Удаление** | `list.Remove(item)` | `db.Table.Remove(item)` + `SaveChangesAsync()` |
| **Масштабируемость** | Ограничена RAM | Гигабайты данных |
| **Транзакции** | Нет | Встроены в EF Core |

## Главные выводы

1. **EF Core — это переводчик между C# и SQL.** Вы пишете LINQ-запросы на C#, а EF Core автоматически преобразует их в оптимальные SQL-запросы. Это позволяет работать с базой данных, не зная глубоко SQL.

2. **Миграции — система контроля версий для структуры БД.** Так же, как Git отслеживает изменения в коде, миграции отслеживают изменения в схеме базы данных. Каждое изменение модели создаёт новую миграцию, которую можно применить или откатить.

3. **Code First удобнее ручного написания SQL.** При изменении структуры данных достаточно изменить C# класс, создать миграцию и применить её к БД. Не нужно вручную писать `ALTER TABLE` или следить за совместимостью изменений.

4. **`SaveChangesAsync()` — ключевой момент.** До вызова этого метода все изменения существуют только в памяти приложения. Только после `SaveChangesAsync()` EF Core отправляет SQL-команды (`INSERT`, `UPDATE`, `DELETE`) в базу данных.

5. **`async/await` при работе с БД — это стандарт, а не опция.** Блокировать поток сервера на время ожидания ответа от базы данных — плохая практика. Асинхронные методы позволяют серверу обрабатывать другие запросы, пока один запрос ожидает базу данных.

6. **LINQ позволяет строить запросы динамически.** Можно добавлять условия `Where()` постепенно, в зависимости от переданных параметров. Запрос уходит в БД только при вызове `ToListAsync()`, `CountAsync()` и т.д. (deferred execution).

7. **Пагинация критически важна для производительности.** Методы `Skip()` и `Take()` преобразуются в `OFFSET` и `LIMIT` в SQL, позволяя загружать только нужную страницу данных, а не все записи из таблицы.

## Как запустить проект

### Требования
- .NET SDK 10.0 или выше
- Git

### Шаги для запуска

```bash
# 1. Клонировать репозиторий
git clone <URL вашего репозитория>
cd Lab31-32_EFCore/TaskDb

# 2. Восстановить пакеты
dotnet restore

# 3. Применить миграции (создать БД)
dotnet ef database update

# 4. Запустить проект
dotnet run
```

После запуска Swagger UI будет доступен по адресу: `http://localhost:5000/swagger`
